The Liturgy Engine: Product & Engineering Vision, Plan, and Delegation Map

How we build rites-as-data under the live product without breaking a single Sunday, then open it outward — phase by phase, brief by brief.

July 20, 2026 · Decisions locked with the owner: announcements in the schema from day one (UI later) · calendar-niche churches are the second segment · LWS-0 lands with strict parity plus an owner-ratified re-baseline escape hatch.

1 · The vision, in one place

Product vision. A church defines its liturgy once — or imports it — as structured data; every week a volunteer turns a date into a family of beautiful print documents in minutes. Customization is structural and textual, never visual. The product's identity is the finish line: typeset, paginated, print-ready worship folders no competitor produces.

Engineering vision. One rite engine renders every document from typed blocks. Calendars and content sources are providers behind narrow interfaces. The current ELCA product becomes an instance of the engine — profile + starter rites + the S&S provider — proven by regression before anything new is built on top. Every phase ships behind the same discipline that built the platform: agent briefs with contracts, my independent gates, sequential merges, full-credential verification.

Prime directive for every brief: if a change is visible to Ascension's Sunday bulletin without the owner deciding it should be, the change is wrong. The 18-test layout suite plus a new golden-PDF parity harness enforce this mechanically.

2 · Core architecture

2.1 The rite schema (LDF-informed, ours)

Rite {
  id, church_id (NULL = library), name, tradition, occasion,   -- "sunday", "funeral"…
  base_rite_id, version, updated_by, updated_at,
  meta: { role_labels: {leader: "P", congregation: "C"},        -- per-church display
          notes },
  blocks: [Block]                                               -- ordered, jsonb
}

Block — common fields:
  { id, type, title?, condition?, toggle? }                     -- toggle: "communion"|"baptism"|…

Block types (v1):
  heading        { text }
  rubric         { text }
  dialogue       { lines: [{role: leader|congregation|instruction|none, text}] }
  literal_text   { text, style: unison|prayer|plain }
  hymn_slot      { slot: gathering|sermon|communion|sending|custom,
                   render: ref|lyrics|notation }
  reading_slot   { slot: first|psalm|second|gospel|custom, render: full|ref }
  psalm          { source: slot|literal, style: responsive|unison }
  proper_slot    { kind: prayer_of_day|confession|offering_prayer|preface|
                         post_communion|blessing|dismissal|…,
                   fallback: text_ref }                         -- into the church text library
  notation       { piece: kyrie|sanctus|agnus_dei|…,            -- resolves via church's setting
                    text_fallback: text_ref }                   -- large-print renders text
  music_item     { kind: prelude|offertory|postlude|choral }
  module_ref     { module_id }                                  -- occasion modules

Reserved now, built later (owner decision):
  prayer_list, week_calendar, serving_list, staff_directory, announcement_text

Condition (Venite-inspired):
  { seasons?: [advent…], feasts?: [...], toggles?: {communion: true}, invert?: bool }

Semantic roles render per the church's role_labels; rubric/instruction is a distinct type; the existing (DialogRole, text) data converts 1:1. Documents differ by renderer style, not by rite: the bulletin renders notation blocks as images, large-print renders their text_fallback — exactly today's behavior, now expressed once.

2.2 Calendar providers

CalendarProvider.resolve(date, church) -> LiturgicalDay {
  date, day_name, season, color,
  cycles: {rcl: "C", narrative: 4, lcms_1yr: true…},   -- multiple simultaneously
  propers: {first: cite, psalm: cite, second: cite, gospel: cite, preface_kind…},
  overlays: [season_of_creation?]
}

Providers: sns (today's behavior, unchanged), rcl_local (owned tables seeded from Vanderbilt data), narrative and lcms_1yr (hand-transcribed owned tables), manual (sermon-series mode: the church types day name + citations; the engine treats them as resolved propers). Per-week override always wins. Seasonal house customs (Lent drops the Kyrie) move out of code into per-church rules keyed by season — evaluated as block Conditions.

2.3 Content sources

ContentSource: get_passage(cite, translation) · get_proper_text(kind, day)
               · get_hymn(collection, number) · capabilities() · entitled(church)

Registry per church: sns (existing client wrapped), public_domain (ELLC texts, WEB/KJV, archived Open Hymnal tunes), net_bible / esv (APIs), church_library (their own saved texts), manual (paste). Resolution per slot: church preference order → entitlement-checked → cached. The S&S entitlement rule generalizes: no source serves a church that isn't entitled to it.

2.4 Storage (migrations 008+)

rites            (id, church_id NULL=library, occasion, name, version, blocks jsonb, meta jsonb, …)
rite_modules     (occasion modules, same shape)
church_texts     (id, church_id, kind, name, body jsonb)        -- the persistent text library
seasonal_rules   (church_id, season, rules jsonb)               -- house customs, ex-season.py
lectionaries     (scheme, cycle_key, date_key, day_name, propers jsonb)  -- owned tables
service_prefs    (existing form_data grows: rite_id, toggles)   -- past_runs unchanged

2.5 What never changes

The renderer owns typography, the auto-tighten pagination system, booklet imposition, and the five-document family. The weekly wizard flow (date → fetch → hymns → generate) is untouched through every phase. No block ever exposes fonts, margins, or layout.

Standing guardrails (inherited from the platform build, plus new ones): agents never touch .env, never push, never change pinned layout numbers except via the escape-hatch process; per-agent test databases; worktree pinning verified as each agent's first action; briefs carry exact contracts and the report-back format; new: any brief touching renderer/templates must run the parity harness and attach its output verbatim; liturgical data transcriptions require a second-pass verification against the source document before merge.

5 · Verification strategy

  • Parity harness (LWS-0a) is built first — golden PDFs across four fixture Sundays chosen to exercise conditions (ordinary, baptism, Lent, festival). Text-layer diff + page-count + pixel-sample. This is the safety net the whole refactor walks on.
  • Existing suites stay authoritative: 570+ fast tests, 18 layout tests, headless browser gates with real S&S credentials (mine to run, agents barred).
  • New suites per phase: schema round-trip + condition evaluation (0b); provider conformance tests shared by all calendar/content providers (3, 4); editor round-trip (2); import corpus regression against the 11 public bulletins (5).
  • The escape hatch, formally: parity gaps that survive a bounded effort produce a before/after PDF pair delivered to the owner; re-baseline happens only on their eyeball approval, recorded in the commit message.
